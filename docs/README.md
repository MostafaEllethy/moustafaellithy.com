# Documentation

This folder contains the foundational documentation for this website. These documents define the site's purpose, structure, visual language, content model, and build plan before any implementation decision is made — they exist so that every future change can be checked against a clear, previously-agreed intent rather than re-litigated from scratch.

They are living documents. When reality and a document diverge, update the document — don't let it go stale.

## Contents

| Document                                       | Describes                                                                                                             |
| ---------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| [`vision.md`](./vision.md)                     | Why the site exists: purpose, mission, audience, positioning, values, and explicit non-goals.                         |
| [`content-strategy.md`](./content-strategy.md) | What content the site contains: section definitions, Writing content pillars, article structure, and metadata format. |
| [`design-system.md`](./design-system.md)       | The visual and interaction language: typography, color, spacing, motion, accessibility, and component principles.     |
| [`architecture.md`](./architecture.md)         | How the site is built: folder structure, routing, content pipeline, and SEO architecture.                             |
| [`roadmap.md`](./roadmap.md)                   | The phased plan for building the site, from vision through launch and ongoing maintenance.                            |

## Suggested reading order

For anyone new to the project — including a future contributor or the site owner returning after time away — the intended reading order is:

1. **`vision.md`** — understand why the site exists before anything else.
2. **`content-strategy.md`** — understand what the site says and how it's organized.
3. **`design-system.md`** — understand how it should look and feel.
4. **`architecture.md`** — understand how it's technically built.
5. **`roadmap.md`** — understand the order of work and what's left to do.

Each document is written to stand on its own, but later documents assume the principles established in earlier ones.
